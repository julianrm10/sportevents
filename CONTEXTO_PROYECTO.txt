=============================================
1. QUÉ ES LA APLICACIÓN
=============================================

EventoSport es una plataforma web de gestión de torneos deportivos. Los usuarios pueden:
- Ver y filtrar eventos deportivos
- Crear equipos y apuntarse a torneos
- Seguir la clasificación en tiempo real
- Guardar eventos favoritos
- Leer noticias deportivas

Los administradores pueden:
- Crear, editar y eliminar eventos
- Generar partidos automáticamente (sistema round-robin)
- Actualizar resultados de partidos
- Finalizar torneos

La aplicación soporta 3 deportes: fútbol, baloncesto y tenis. Cada uno tiene sus propias columnas de clasificación (goles/puntos/sets).

=============================================
2. STACK TECNOLÓGICO
=============================================

- Backend:    Node.js + Express 4
- Vistas:     EJS (template engine, server-side rendering)
- Base datos: MySQL con mysql2 (pool de conexiones, promise-based)
- Estilos:    Bootstrap 5 + CSS propio (public/css/style.css)
- Iconos:     Bootstrap Icons (bi bi-*) + Tabler Icons (ti ti-*)
- Auth:       JWT almacenado en cookie httpOnly (7 días), bcrypt para passwords
- Sesión:     express-session + connect-flash (mensajes flash)
- Uploads:    multer → public/uploads/ (jpg/png/gif/webp, máx 5MB)
- Noticias:   API externa GNews (fetch en /noticias)
- Docs:       paquete docx (generación de Word, sin uso activo visible en vistas)
- Dev:        nodemon (script dev), seed.js para datos de prueba

El servidor arranca en el puerto 3000 (o env PORT). Entry point: app.js.

6. AUTENTICACIÓN Y SESIÓN
=============================================

- Al hacer login se crea un JWT firmado con JWT_SECRET (expira 7d).
- Se almacena en cookie "token" (httpOnly, sameSite: lax).
- El middleware global de app.js lo verifica en cada request:
  - Si válido: inyecta req.user = { id, nombre, email, role } y res.locals.user
  - Si inválido/expirado: limpia la cookie y asigna null
- requireAuth (middlewares/auth.js): si no hay req.user → redirige a /auth/login
- requireAdmin (middlewares/admin.js): si req.user.role !== 'admin' → 403

En las vistas EJS, el usuario está disponible como la variable "user" (puede ser null si no autenticado).

8. LÓGICA DE NEGOCIO IMPORTANTE
=============================================

CLASIFICACIÓN (round-robin):
- Se calcula en tiempo real en eventController.js::eventDetail()
- Para cada partido jugado se suman puntos a ambos equipos
- Victoria = 3 puntos, Empate = 1 punto (no aplica en tenis), Derrota = 0
- En tenis: no hay empate (score_team1 nunca = score_team2 si jugado)
- Orden: puntos DESC → diferencia (GF-GC) DESC → GF DESC
- Las columnas de la tabla cambian nombre según el deporte:
  · fútbol:     GF, GC, DG (Goles a favor, Goles en contra, Diferencia de goles)
  · baloncesto: PF, PC, DP (Puntos a favor, Puntos en contra, Diferencia de puntos)
  · tenis:      SF, SC, DS (Sets a favor, Sets en contra, Diferencia de sets)

GENERACIÓN DE PARTIDOS (round-robin):
- Se generan todos los enfrentamientos posibles: n*(n-1)/2 partidos
- Todos con score 0-0, estado 'pendiente'
- El evento pasa a estado 'en_curso' automáticamente
- Se usa una transacción MySQL (connection.beginTransaction)

INSCRIPCIÓN DE EQUIPOS:
- El equipo debe pertenecer al usuario autenticado
- El tipo del equipo debe coincidir con el tipo del evento
- Un usuario solo puede inscribir un equipo por evento (UNIQUE en registrations)
- Si ya tiene equipo inscrito y selecciona otro: se hace UPDATE (cambio de equipo)
- No se puede inscribir si el evento ya alcanzó max_equipos (a menos que ya esté inscrito)

CANCELAR INSCRIPCIÓN:
- Solo si el evento sigue en estado 'abierto'
- Solo disponible desde /perfil (botón en tab inscripciones)

ELIMINAR EQUIPO:
- No se puede eliminar un equipo que esté inscrito en un evento en_curso
- Si se elimina: primero se borran las registraciones del equipo, luego el equipo

FINALIZAR EVENTO:
- Solo si estado = 'en_curso'
- Solo si pendingCount = 0 (todos los partidos tienen estado 'jugado')

10. CONVENCIONES DE CÓDIGO
=============================================

- Arquitectura MVC: routes → controllers → models
- Los modelos devuelven siempre el resultado de mysql2 (array destructured: [rows])
- Los controllers hacen res.render() o res.redirect() directamente
- Mensajes flash: req.flash('success'/'error', 'mensaje') antes de redirect
- En las vistas EJS se accede a: user (objeto o null), eventos, event, teams, etc. según la vista
- Las rutas de acción POST que modifican datos redirigen después (PRG pattern)
- Los modales de confirmación usan el partial modal-confirm.ejs con Bootstrap
- Iconos de deporte en avatares: ti ti-ball-football (fútbol), ti ti-ball-basketball, ti ti-ball-tennis
- Colores por deporte: fútbol=success(verde), baloncesto=warning(naranja), tenis=info(azul)
- Clases CSS por deporte: .team-card-futbol, .sport-bg-futbol, etc.

11. COSAS A TENER EN CUENTA
=============================================

- El campo "tipo" en events y teams usa los valores: 'futbol', 'baloncesto', 'tenis'
  (antes era 'futbol_sala', se cambió a 'futbol' — si hay datos en BD hay que migrarlos:
   UPDATE events SET tipo='futbol' WHERE tipo='futbol_sala';
   UPDATE teams  SET tipo='futbol' WHERE tipo='futbol_sala';)

- Las imágenes de eventos se guardan en public/uploads/ con nombre timestamp_random.ext
  Al editar o eliminar un evento se borra el archivo del filesystem con fs.unlink

- La navbar muestra el enlace /admin solo si user.role === 'admin'

- En /perfil el flag "en_curso_activo" en cada equipo indica si tiene inscripción
  activa en un evento en estado 'en_curso' (para deshabilitar botón de eliminar)

- connect-flash requiere que express-session esté configurado antes

- El token JWT se decodifica en el middleware global de app.js y se inyecta en
  res.locals.user, por lo que todas las vistas EJS tienen acceso a "user" directamente
